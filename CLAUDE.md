# Mstream iOS — AI Agent Guide

Native iOS WKWebView wrapper for Cineby (movies/series) and Nimegami (anime). All logic resides in a **single file**: `ios-cineby/WebViewController.swift` (~1300 lines).

---

## Project Structure

```
ios-cineby/              ← The ONLY folder that needs editing
  WebViewController.swift  ← Complete app logic (Swift + JS injection)
  OrientationNavigationController.swift
  SceneDelegate.swift
  AppDelegate.swift
capacitor-app/
  patch_ios.sh           ← Script to merge source into Xcode project
  src/                   ← Web assets (not used for iOS)
```

> **Do not touch** `capacitor-app/ios/` — this folder is auto-generated by `patch_ios.sh` and will be overwritten during build.

## Architecture: `WebViewController.swift`

### Native Swift (UIKit)

| Component | Description |
|---|---|
| `setupWebView()` | Configures WKWebView + injects JS via `WKUserScript` |
| `setupLandingView()` | Portal selection screen (Cineby / Nimegami) |
| `setupSwitchWebButton()` | Floating button top-right, toggles portal |
| `setupPortraitRotateButton()` | Appears only when video is detected (portrait) |
| `setupLandscapeRotateButton()` | Appears in landscape mode |
| `setupNativeLockButton()` | Netflix-style playback lock (landscape only) |
| `setOrientationVisual()` | Visual rotation of WebView via `CGAffineTransform` (not system rotation) |
| `handleScreenTap()` | Toggles lock button visibility when screen is tapped |
| `broadcastPlaybackLockState()` | Sends postMessage to all iframes |
| `userContentController()` | Receives messages from JS: `hasVideo`, `triggerRotation`, `frameLoaded` |

### Injected JavaScript (inside `setupWebView()`)

The script is injected into **all frames** (`forMainFrameOnly: false`), including player iframes. Key functions:

| JS Function | Purpose |
|---|---|
| `forcePlaysInline()` | Sets `playsinline` so the video plays inline rather than forcing native fullscreen |
| `injectMstreamControls()` | Injects playback overlay (Back 5s / Play / Forward 5s) into `document.body` with `position:fixed` |
| `hideDefaultControls()` | Hides default player control bar via specific CSS classes (no wildcards) |
| `broadcastActiveSite()` | Notifies subframes about the active site name |
| `handleGlobalTouch` | Global capture-phase listener: tap anywhere → show overlay |
| `playbackLock` message | Recursively locks/unlocks all default player controls across all iframes |

### Dedicated Overlay (`#mstream-controls-overlay`)

- Appended to `document.body` with **`position: fixed`** — guaranteed to render on top of all player libraries
- `freezePlayerInterceptors()`: When a button is touched, temporarily disables `pointer-events` on video ancestors (up to 6 levels) so the player does not steal the touch
- `unfreezePlayerInterceptors()`: Restores `pointer-events` on `touchend`
- Play/pause state is synchronized every 500ms via `setInterval`

## Video Detection Flow

```
JS setInterval (1s) → checkVideoPresence()
  → postMessage { hasVideo: true/false }
    → Swift: userContentController()
      → updateRotateButtonVisibility()
        → portraitRotateButton.isHidden = !hasVideo
```

## CSS Hiding Strategy

Use **specific class names** per player library — DO NOT use wildcard selectors like `[class*="..."]` because they might hide the video rendering layer, resulting in a black screen, or block our own overlay.

Supported players: JW Player, Video.js, Plyr, Artplayer, DPlayer, Shaka Player.

CSS always contains an explicit exception rule:
```css
#mstream-controls-overlay, #mstream-controls-overlay * {
    display: flex !important;
    pointer-events: auto !important;
}
```

## AI Working Rules

1. **Always edit `ios-cineby/WebViewController.swift`** — this is the single source of truth.
2. **Do not create new Swift files** — simply modify the existing ones.
3. **Do not add dependencies** (CocoaPods/SPM) — the project is intentionally zero-dependency.
4. **When modifying CSS**: Always use specific class names, not wildcards. Always maintain the `#mstream-controls-overlay` exception.
5. **When modifying JS**: Scripts are injected into **all frames**, side effects can occur in third-party player iframes. Mentally test on both sites (Cineby via iframe embed, Nimegami via direct video).
6. **Nimegami**: Video is directly in the main frame. **Cineby**: Video is inside third-party player iframes (vidsrc, etc.).
7. **Do not read the entire file** if you only need to modify one section — use `view_file` with a specific range or use `grep_search` first.
8. **Flexibility to edit other files**: If a problem requires modifying files other than `WebViewController.swift` (e.g. `SceneDelegate.swift`, `AppDelegate.swift`, or `CLAUDE.md`) for correct functionality, the AI **can and must** edit them without requesting extra permission. Correct behavior is the priority, not scope limitations.

## Debugging Guide Specific to Cineby

- **Lock does not hide playback?** → Check if `broadcastPlaybackLockState()` is called for Cineby (it should be called for all sites). Cineby uses third-party iframes that must receive the `playbackLock` postMessage.
- **CSS lock not effective in iframe?** → Use `hideCinebyNativeControls()`, which walks through frames using `walkFrames()` to inject CSS directly into every accessible frame.
- **Cineby player not detected?** → The video is in a cross-origin iframe (vidsrc, etc.). `earlyScript` is already injected into all frames, so `postMessage` from the iframe to Swift should work.
